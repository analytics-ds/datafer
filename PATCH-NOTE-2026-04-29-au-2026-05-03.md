## Récap rapide

### API

- Endpoint progress polling : `/api/briefs/[id]/progress` pour suivre l'analyse en temps réel depuis un script externe
- Auth via session web (cookie Better-auth) OU Bearer API key, transparent pour l'API

### Améliorations
- **Crawler 3 niveaux** : fetch direct → Bright Data Web Unlocker → Bright Data Scraping Browser (Puppeteer/CDP). Plus de pages bloquées sur Nike, Wethenew, Cloudflare, Datadome
- **Provider CrazySerp** par défaut (~150x moins cher que SerpAPI) avec clé fallback auto si la primaire est épuisée
- **Locale SERP** correctement forcée par pays (gl + hl + googleDomain). Avant : "tshirt blanc" en FR ramenait du H&M en US
- **Position client jusqu'au top 100** : déclenchement automatique d'un scrape étendu quand le client n'est pas trouvé dans le top 17
- **Crawl URL utilisateur en parallèle des concurrents** : gain de 20 à 30 secondes par brief
- **Hiérarchie préservée à l'export** : titres H1 à H6, paragraphes, listes ul/ol/li, tableaux, blockquote, gras/italique inline
- **Score SEO refondu** (itération 5) : NLP coverage split Essentiels (15 pts) + Importants (10 pts), GEO réintégré à 5 % du score
- **Matching keyword tolérant** aux flexions FR (chaussure/chaussures, basket/baskets) et aux accents (côte/cote/coté), avec dédup automatique
- **Loader brief plus parlant** : 5 étapes en live (SERP → crawl X/10 sites → NLP → scoring → saving)

### Corrections automatiques (auto-recovery)
- **Zéro brief zombie** : Cloudflare Queue + worker consumer dédié + cron de cleanup toutes les minutes. Tout brief pending au-delà de 2 min est forcé en `failed` automatiquement, worst-case 3 min visibles utilisateur
- **Fallback automatique CrazySerp** : si la clé primaire répond 0 résultats ou plante (quota épuisé), bascule auto sur la clé secondaire pour le brief en cours
- **Retry myUrl auto** : si Bright Data renvoie une erreur transitoire (5xx/429) sur le crawl de l'URL utilisateur, retry automatique 1 fois avant abandon
- **Race condition cron / consumer fixée** : un brief qui finit pile au moment du cleanup ne voit plus son résultat valide écrasé en `failed`
- **Briefs stuck en pending** sur les keywords difficiles : timeouts ajustés pour rester sous le wall Cloudflare Workers
- **Score liste vs détail** : valeurs cohérentes maintenant entre la card et la vue brief
- **Spellcheck FR auto** dans l'éditeur : correction orthographique navigateur active dans la langue du brief, sans config

### Sécurité (audit complet du 3 mai)
- **XSS sanitisé** dans les exports HTML et PDF partagés publiquement : whitelist stricte de tags + attributs, drop des `<script>/<iframe>/<style>`, neutralisation des `javascript:` et `data:` dans les `href`, strip des handlers `on*`
- **Fail-fast sur secret manquant** : le worker refuse de démarrer si `BETTER_AUTH_SECRET` n'est pas configuré (avant : sessions signées avec une clé vide)

## Détail

### Création de brief

#### Mode Batch (nouveau)
- Toggle `1 brief` / `Batch (jusqu'à 5)` en haut du formulaire.
- En batch, on démarre avec une seule ligne, bouton `+ Ajouter un brief` pour empiler jusqu'à 5.
- Chaque row a son **keyword + URL existante + pays + client**, indépendants.
- Le Brief 1 sert de **valeur par défaut** : si tu remplis pays/client uniquement pour le Brief 1, les briefs suivants en héritent (placeholder visuel `↑ FR`, `↑ Celio (Brief 1)`). Tu peux override row par row.
- Croix par row pour supprimer.
- Au submit, redirection vers le dossier client si tous les briefs partagent le même client, sinon vers la liste globale.

#### Mode Simple (préservé)
- Interface initiale conservée pour la création d'un brief unique avec son polling de progression en 5 étapes (SERP → crawl → NLP → scoring → saving).

#### URL existante
- Crawl de la page utilisateur en parallèle des concurrents (gain de 20 à 30 secondes), retry 1 fois si échec Bright Data transitoire.
- Contenu injecté tel quel dans l'éditeur (préserve la hiérarchie H1/H2/P, gras, italique, listes, tableaux).
- Score initial calculé sur ce contenu vs SERP top 10.

### Crawler

#### Cascade 3 niveaux
1. **Fetch direct** (gratuit, ~70 % de succès), User-Agent Googlebot smartphone à jour.
2. **Bright Data Web Unlocker** (REST, ~$1.50/CPM). Résout Cloudflare, Datadome, PerimeterX. Zone `web_unlocker1` Premium activée depuis le 2 mai.
3. **Bright Data Scraping Browser** (WSS Puppeteer/CDP, ~$8/GB) pour les pages JS-only type Nike Snkrs, Wethenew. Client CDP raw écrit à la main pour passer le scheme https + Basic auth header car Cloudflare Workers ne supporte ni `wss://` dans fetch ni les credentials inline.

#### Détection robuste
- Détecte les pages "challenge" Cloudflare même servies via Browser Rendering.
- ScrapingBee retiré définitivement (remplacé par Bright Data).
- Timeouts ajustés pour rester sous le wall Cloudflare Workers.

#### Parser
- `parseHTML` extrait le contenu éditorial uniquement (filtre nav/footer/sidebar/menus).
- `structuredHtml` préserve H1 à H6, paragraphes, listes (ul/ol/li), tableaux, blockquote, gras et italique inline.
- Filtres "noise" affinés : retrait de `modal-*`, `popup-*`, `product-card` (qui bouffaient le contenu sur Nike, Shopify, etc.).

### SERP

#### Provider switchable
- `SERP_PROVIDER = "crazyserp"` par défaut. Bascule sur SerpAPI possible en éditant `wrangler.toml`.
- **Clé fallback** : si la primaire répond 0 résultats ou plante, bascule auto sur la fallback pour le brief en cours. Logs : `[crazyserp] primary key failed, trying fallback`.
- Locale forcée par pays : `gl + hl + googleDomain` envoyés explicitement (sinon SERP US biased).

#### Position client jusqu'au top 100
- Avant : on ramenait ~17 résultats max via CrazySerp, donc les clients positionnés rang 18+ remontaient `position N/A`.
- Maintenant : si le brief a un client lié et que la position n'est pas trouvée dans le top 17, déclenchement automatique d'un appel `page=10` CrazySerp (cumulatif, ~100 résultats organiques, 10 crédits FR ≈ $0.005).
- Surcoût payé uniquement pour les briefs avec client lié ET site non rangé top 17. Pas d'impact sur les briefs sans client ou clients déjà top 10.

### Architecture asynchrone

#### Cloudflare Queue + worker consumer
- 2 workers : `datafer` (producer Next.js) et `datafer-analysis-consumer` (dédié à l'analyse).
- À la création d'un brief, le producer enqueue un message dans `datafer-analysis` puis répond 200 immédiatement (~100 ms).
- Le consumer récupère le message, fait crawl + NLP + scoring + écriture en base. Budget CPU/wall isolé du HTTP.
- Config consumer : `max_batch_size = 1`, `max_retries = 1`, DLQ `datafer-analysis-dlq`.

#### Cron anti-zombie
- GitHub Actions cron `* * * * *` (latence GHA 1-5 min réelle).
- Tout brief `pending` depuis plus de 2 min est forcé en `failed`.
- Worst-case visible utilisateur : 3 min entre crash worker et passage en failed.
- **Race condition fixée** : le consumer fait son `UPDATE status='ready'` conditionné sur `WHERE status='pending'` pour ne pas écraser un `failed` du cron arrivé entre-temps.

### Scoring SEO

Pondération sur 100 (itération 5) :

| Critère | Max | Détail |
|---|---|---|
| keyword | 15 | softScore (couverture tokens) max 7 + bonus exact max 8, tolérance flexions FR + accents |
| nlpCoverage | 25 | Split par tier : Essentiels (presence ≥ 70) → 15 pts linéaire + Importants (40-70) → 10 pts linéaire |
| contentLength | 12 | wc dans `[minWordCount, maxWordCount]` → 12 |
| headings | 15 | H1/H2/H3 + keyword dans H1/H2 |
| placement | 15 | KW exact ou soft dans first 100 mots / 1ère phrase / H1 / H2 |
| structure | 9 | Listes, tableau, FAQ, etc. |
| quality | 9 | Diversité lexicale, longueur phrases, density |

Total SEO weight = 0.95, GEO weight = 0.05.

#### GEO (Generative Engine Optimization)
- Checklist 5 pts intégrée au score : tableau structuré, liste à puces (≥3 items), TL;DR, section FAQ (≥2 questions), données chiffrées (≥3, regex unités €/$/%).

#### Matching keyword
- Tolérance flexions françaises (chaussure / chaussures, basket / baskets) avec dédup.
- Normalisation des accents avant matching.
- Tolère jusqu'à 2 mots interposés entre les tokens du keyword.
- Soft match cohérent entre liste et détail brief.

### Sécurité

Audit complet du 3 mai. Fixes en prod :

#### XSS dans exports HTML / print
Le contenu de l'éditeur (`editorHtml`) était injecté brut dans les exports. Exploitable via les endpoints publics de partage.

Fix : sanitiseur HTML maison via `htmlparser2`. Whitelist stricte de tags + whitelist d'attributs par tag. Drop complet des tags dangereux (`<script>`, `<style>`, `<iframe>`, `<object>`, `<embed>`). Neutralise `javascript:`, `data:` dans les `href`. Strip tous les `on*` handlers et `style` inline. Auto `rel="noopener noreferrer"` sur `target="_blank"`.

#### Fail-fast sur `BETTER_AUTH_SECRET` manquant
Avant : si le secret n'était pas configuré, `betterAuth` recevait `undefined` et signait les sessions avec une clé vide. Maintenant : `throw` au boot.

#### Race condition cron / consumer
Le cron passait les briefs >2min en `failed`. Si le consumer terminait pile à ce moment, il pouvait écraser le `failed`. Fix : tous les UPDATE conditionnés sur `WHERE status='pending'`.

#### Traçabilité API key
`resolveUser` retournait `email: ""` pour les requêtes API key. Maintenant : join sur `user.email`, l'email réel apparaît dans les logs.

#### Workspace partagé (assumé)
Décision : workspace partagé entre l'équipe Datashake. Toute personne authentifiée peut lire/éditer/supprimer/partager n'importe quel brief ou dossier. Comportement volontaire.

### Stack et infra

- Next.js 16 + React 19 + OpenNext Cloudflare adapter → Cloudflare Workers.
- Cloudflare D1 (SQLite) via Drizzle ORM.
- Cloudflare Queue + DLQ pour l'analyse asynchrone.
- Better-auth (email + password, `disableSignUp: true`).
- Bright Data : Web Unlocker REST + Scraping Browser CDP.
- CrazySerp (provider SERP par défaut) + SerpAPI (fallback config).
- Haloscan (volume, CPC, KGR, questions complémentaires).

#### Déploiement
- Pas d'auto-deploy. `npm run deploy` (worker producer) + `npx wrangler deploy --config wrangler-analysis.toml` (worker consumer) à lancer manuellement.
- Migrations D1 : `npm run db:generate` puis `npm run db:migrate:remote` avant déploiement de code qui dépend des nouvelles colonnes.
